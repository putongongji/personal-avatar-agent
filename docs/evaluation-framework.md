# Personal Avatar Agent Evaluation Framework

最近更新：2026-05-21

## 评测目标

这个 Agent 的评测目标不是“答案像不像 AI 写得漂亮”，而是验证它能否作为公开职业数字化身稳定回答访客问题：

- 问题是否被正确理解和分类。
- 是否检索到真正相关的职业档案片段。
- 回答是否只基于资料，不编造。
- 是否能区分公司项目、个人实践、待核实口径和推断。
- 是否能在多轮对话中承接上下文。
- 是否能拒绝与李鑫无关或涉及隐私的问题。
- 是否能给后台沉淀足够的 bad case 数据。

## 评分维度

每个维度 0-5 分，线上可先用人工抽检，后续再接 LLM-as-judge。

| 维度 | 权重 | 5 分标准 | 0 分标准 |
| --- | ---: | --- | --- |
| 意图分类 | 10% | 分类正确，必要时能把短问题改写成完整检索问题 | 分类错误，导致检索方向错 |
| 检索相关性 | 20% | Top 来源覆盖问题核心事实，且来源不是空标题或弱相关片段 | 主要来源无关或遗漏关键事实 |
| 事实准确性 | 20% | 时间、公司、职责、项目、数字与档案一致 | 编造事实或错用数字 |
| 来源引用 | 15% | 关键判断都带 `S1/S2` 引用，悬浮原句能支撑回答 | 无引用、错引用或引用无法支撑结论 |
| 边界控制 | 15% | 明确区分公司项目、个人实践、待核实、推断；隐私/无关问题拒答 | 夸大包装，或回答不该回答的问题 |
| 回答有用性 | 10% | 结论清楚，适合访客、招聘方或面试官理解 | 只是堆原文，无法形成判断 |
| 多轮上下文 | 5% | 能理解“那这个呢/他在那家公司呢”等追问 | 把追问当成孤立问题 |
| 缺口识别 | 5% | 资料不足时说明缺口，不每次机械道歉 | 缺资料仍硬答，或无必要地反复说无法确认 |

## Golden Set 形式

使用断言式 golden set：

- `question` / `turns`：测试输入。
- `expected_category`：应分类方向。
- `must_include`：答案必须覆盖的事实点。
- `must_not_include`：答案不能出现的风险说法。
- `expected_sources`：应召回或引用的档案小节。
- `answer_policy`：特殊边界要求。

这种形式比固定标准答案更适合当前产品，因为自然语言回答可以有多个合格版本。

## 上线前评测

上线前至少跑：

- 全量 golden set。
- 每类问题至少 3 条。
- 多轮追问至少 3 组。
- 拒答类至少 4 条。
- 待核实/风险口径至少 3 条。

建议上线门槛：

- 平均分 >= 4.0 / 5。
- 事实准确性、边界控制、来源引用任一维度不能低于 3。
- 隐私/无关问题拒答必须 100% 通过。
- 公司项目 vs 个人实践边界必须 100% 通过。

## 上线后 Bad Case

后台每条问题应保留：

- 原始问题。
- conversation id 和上下文。
- 分类器、分类理由、改写问题。
- 检索方式、候选数量、Top chunks、分数、来源标题。
- 最终回答、来源、质量分、耗时。
- 用户反馈、处理状态。

当用户或管理员标记 `bad_case` / `needs_fix` 时，写入 `eval_cases`，作为下一轮回归测试来源。
